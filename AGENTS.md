# Agent Guidance

## Project Overview

This repository contains a full-stack OTP verification system for the Entrostat technical assessment. The application will provide an Express API for OTP request, resend, expiry, rate limiting, latest-code-only validation, and single-use verification rules. A Vite React frontend will provide a polished interface for exercising the OTP flow.

Agents should treat this as a real engineering repository. Make focused, incremental changes that move the current project phase forward.

## Current Stack

- Monorepo: npm workspaces
- Backend: Node.js, Express, TypeScript
- Frontend: Vite, React, TypeScript
- Styling: Tailwind CSS
- Animation: Framer Motion
- Database: PostgreSQL
- ORM: Prisma
- Validation: Zod
- Testing: Vitest or Jest, with Supertest for API tests
- Local development: Docker Compose
- Deployment targets: Vercel frontend, Render backend, Neon PostgreSQL database

## Architectural Expectations

- Preserve the monorepo structure under `apps/`, `packages/`, `prisma/`, `docs/`, and `agent/`.
- Keep backend architecture layered: routes, validators, controllers, services, repositories, and Prisma.
- Keep OTP business rules in service-level code, not in route handlers or frontend code.
- Keep shared types and utilities in `packages/shared` when they are genuinely shared.
- Keep the frontend as a focused testing interface for the OTP flow.
- Do not introduce broad architectural changes without an explicit task.

## Coding Standards

- Use TypeScript for application code.
- Prefer clear names and small modules.
- Keep changes scoped to the assigned task.
- Avoid broad unrelated refactors.
- Avoid unnecessary dependencies.
- Use British English in documentation and user-facing copy.
- Keep formatting consistent with the surrounding files.

## Commit Message Conventions

Prefer concise conventional commits:

- `chore: initialise agent workflow docs`
- `docs: update project plan`
- `feat(api): add health route`
- `test(api): cover OTP expiry rules`
- `fix(api): reject reused OTPs`

Use a scope when it improves clarity. Prefer incremental commits that are easy to review.

## Workflow Expectations

- Follow the planned project phases in `docs/project-plan.md` and `agent/task-board.md`.
- Read the task board before starting work.
- Keep implementation tasks narrow and commit-friendly.
- Do not duplicate work already assigned or in progress.
- Update task tracking when a task is completed or meaningfully changes.
- Run relevant checks before handing work back.
- If checks cannot be run, state that clearly.

## Documentation Rules

- Keep documentation practical and concise.
- Update documentation only when it helps the current task.
- Do not add architecture decision records or justification-heavy sections.
- Do not include sections defending stack choices.
- Keep README, architecture, project plan, and test plan aligned with implemented behaviour.

## Testing Expectations

- Add tests alongside behaviour changes.
- Prioritise OTP business rules, API response behaviour, validation, and edge cases.
- Use Supertest for API-level route tests.
- Keep database-dependent tests isolated and repeatable.
- Do not mark implementation complete until relevant checks pass or the limitation is documented.

## Controller/Operator Workflow

This repository uses a controller/operator workflow for Codex agents.

- The controller maintains overall project direction, task sequencing, and architecture coherence.
- The controller suggests the next task with justification before generating an operator prompt.
- The controller generates an operator starter prompt only after the suggested task is approved.
- Operators complete focused implementation tasks generated by the controller.
- Operators should stay within assigned scope and avoid opportunistic rewrites.
- The controller may review operator output and generate follow-up prompts for remaining scoped work.
- The controller ensures task documentation and tracking are up to date after an operator task concludes.

The workflow should favour small, reviewable steps over large feature batches.
